    Сценарии развёртывания дата-центра¶

    Практическая инструкция для сетевого инженера: как настраивать коммутаторы в дата-центре под разные сценарии с безопасным циклом изменений.

    1. Базовый безопасный процесс (для любого сценария)¶

    1. Открыть сессию и перейти в режим конфигурации.
    2. Внести set/delete изменения.
    3. Выполнить проверки до применения.
    4. Применить через commit confirmed, затем подтвердить.
    5. Сохранить в startup.

    Рекомендуемый шаблон:

    configure
... команды set/delete ...
validate
compare candidate running
commit dry-run
commit confirmed 10
show running
confirm
save
end

    Если после commit confirmed не сделать confirm, конфигурация будет автоматически откатана.

    2. Сценарий A: Leaf L2 Access (серверный VLAN)¶

    Когда использовать: - стойка серверов, VLAN access на портах leaf-коммутатора.

    Шаги:

    configure
set bridges bridge br10 create
set vlans vlan 10 name SERVERS
set interfaces swp1 bridge br10
set interfaces swp1 l2 mode access vlan 10
set interfaces swp2 bridge br10
set interfaces swp2 l2 mode access vlan 10
validate
commit confirmed 10
show vlans
show bridges
confirm
save
end

    Проверки: - show vlans — VLAN создан и назначен. - show bridges — интерфейсы в нужном bridge.

    3. Сценарий B: Spine/Leaf underlay (eBGP L3 fabric)¶

    Когда использовать: - L3 underlay между spine и leaf.

    Пример для узла:

    configure
set interfaces swp1 ipv4 address 192.0.2.1/31
set interfaces swp2 ipv4 address 192.0.2.3/31
set routing bgp local-as 65001
set routing bgp router-id 192.0.2.1
set routing bgp neighbor 192.0.2.0 remote-as 65002
set routing bgp neighbor 192.0.2.2 remote-as 65003
set routing bgp neighbor 192.0.2.0 next-hop-self
set routing bgp neighbor 192.0.2.2 next-hop-self
validate
commit confirmed 10
show bgp summary
show ip route
confirm
save
end

    Проверки: - show bgp summary — сессии в состоянии Established. - show ip route — маршруты underlay присутствуют.

    4. Сценарий C: EVPN/VXLAN multi-tenant¶

    Когда использовать: - overlay L2/L3 между leaf-коммутаторами для tenant-сетей.

    Пример:

    configure
set routing bgp local-as 65001
set routing bgp router-id 192.0.2.10
set evpn vtep source-ip 192.0.2.10
set vlans vlan 10 name TENANT10
set evpn vlan 10 vni 10100
set evpn l3-vni 5000 vrf BLUE
set evpn rd 65001:5000
set evpn rt import 65001:5000
set evpn rt export 65001:5000
set vrf BLUE rd 65001:5000
set vrf BLUE interface swp10
set interfaces vlan 10 ipv4 address 10.10.10.1/24
set routing bgp evpn neighbor 198.51.100.2 activate
validate
commit confirmed 10
show evpn
show vrf
show bgp l2vpn evpn summary
confirm
save
end

    Проверки: - show evpn — VNI/RT/RD применены. - show bgp l2vpn evpn summary — EVPN-соседство поднято.

    5. Сценарий D: MLAG pair + агрегирование uplink/downlink¶

    Когда использовать: - два leaf в паре MLAG, подключение серверов/аплинков через LACP.

    Пример:

    configure
set mlag domain FABRIC1 peer-link po10 keepalive 192.0.2.100
set port-channel id 10 members swp49,swp50 mode active
set interfaces swp49 channel-group id 10 mode active
set interfaces swp50 channel-group id 10 mode active
validate
commit confirmed 10
show mlag
show port-channel
confirm
save
end

    Проверки: - show mlag — домен и peer-link активны. - show port-channel — участники в bundle.

    6. Сценарий E: базовые сервисы управления и доступа¶

    Когда использовать: - первичная подготовка коммутатора (hostname, NTP, AAA).

    configure
set system hostname leaf01
set system ntp server 10.0.0.10
set system dns nameserver 10.0.0.53
set aaa role readonly permit show
set aaa role readonly permit health_check
set aaa user noc password <strong-password> role readonly
validate
commit confirmed 10
show system
show aaa
confirm
save
end

    Проверки: - show system — hostname/NTP/DNS применены. - show aaa — роль и пользователь видны в конфигурации.

    7. Сценарий F: EVPN Type-5 (IP Prefix Route) для L3 overlay¶

    Когда использовать: - нужно рекламировать L3-префиксы между VTEP/VRF через EVPN (Type-5).

    Пример:

    configure
set vrf BLUE rd 65001:5000
set vrf BLUE route-leak export 65001:5000
set vrf BLUE route-leak import 65001:5000
set evpn l3-vni 5000 vrf BLUE
set evpn rd 65001:5000
set evpn rt import 65001:5000
set evpn rt export 65001:5000
set routing bgp evpn neighbor 198.51.100.2 activate
validate
commit confirmed 10
show bgp l2vpn evpn summary
show evpn vni detail
show vrf
confirm
save
end

    Проверки: - show bgp l2vpn evpn summary — EVPN AF в Established. - show evpn vni detail — L3VNI в рабочем состоянии. - show vrf — VRF и его привязки присутствуют.

    8. Сценарий G: Symmetric IRB + Anycast Gateway¶

    Когда использовать: - inter-VLAN routing в EVPN fabric с единым шлюзом для хостов на разных leaf.

    Пример:

    configure
set evpn irb-mode symmetric
set evpn anycast-gateway mac 00:1c:73:00:00:99
set vrf BLUE rd 65001:5000
set evpn l3-vni 5000 vrf BLUE
set vlans vlan 10 name TENANT10
set evpn vlan 10 vni 10100
set interfaces vlan 10 virtual-gateway-address 10.10.10.1/24
set vrf BLUE interface vlan10
validate
commit confirmed 10
show evpn
show vrf
show ip route
confirm
save
end

    Проверки: - show evpn — IRB/Anycast и VNI параметры применены. - show vrf — VLAN-интерфейс привязан к VRF. - show ip route — маршруты меж-VLAN видны.

    9. Безопасное удаление конфигурации (важный порядок)¶

    При деинсталляции VLAN/портов придерживайтесь порядка:

    1. Снять L2-настройки с интерфейса.
    2. Отвязать интерфейс от bridge.
    3. Удалить VLAN.
    4. При необходимости удалить bridge.

    Пример:

    configure
delete interfaces swp1 l2
delete interfaces swp1 bridge
delete vlans vlan 100
delete bridges bridge br10
validate
commit confirmed 10
confirm
save
end

    Это снижает риск preflight-ошибок вида "интерфейс имеет L2 режим, но не привязан к bridge".

    10. Операционный чек-лист перед/после change¶

    Перед change: - есть окно работ и rollback-план; - собран бэкап (show running, show startup); - проверен текущий статус (health check, show blockers).

    После change: - show running соответствует intended state; - все контрольные show команды по сценарию зелёные; - выполнен save; - зафиксированы изменения и результаты в change log.

    11. Быстрый troubleshooting (если commit не проходит)¶

    Проверить: - validate и show blockers до повторного commit. - compare candidate running для точного diff. - commit dry-run для предпросмотра пайплайна применения.

    Типовые причины: - интерфейс в L2-режиме не привязан к bridge; - VRF/VNI созданы, но не связаны друг с другом; - BGP EVPN сосед не активирован (set routing bgp evpn neighbor <ip> activate); - пропущен confirm после commit confirmed (авто-rollback по таймеру).
